# SyncCast 迭代需求报告（2026-05-16）

## 1. 当前事实

- 本地 Stereo 是稳定产品核心；用户已确认本地 Stereo 正常、屏幕休眠恢复问题已解决。
- Whole-home / AirPlay 已能让所有选中设备出声，但 Local + AirPlay 的自动同步仍未解决。问题不是两个 AirPlay 设备彼此同步，而是本地 Mac/显示器/CoreAudio 输出与 AirPlay 组之间的可变延迟。
- 固定手动 delay 数值没有长期意义；AirPlay 建组、设备切换、音量变化、中断恢复后，当前 route 的相对延迟可能变化。
- 高频主动 probe 在真实设备上仍可能被听见。主动声学诊断必须保持 lab-only；默认自动路线应优先无探针、正常播放、被动观测。
- 本地 Stereo 默认 Direct Stereo，理论上绕开 ScreenCaptureKit；Process Tap 已存在但 DRM 与 AirPlay capture-dependent 路径仍需验证。

## 2. 需求优先级

### P0：保护稳定 Stereo

不能为了 AirPlay 校准牺牲本地 Stereo。

验收：
- Stereo 两个本地输出继续稳定播放。
- 屏幕休眠/唤醒后自动恢复。
- 默认 Stereo 不启动 SCK，不触发 Screen Recording 语义。
- 任何失败都必须恢复或保留用户原有输出状态。

### P1：Local + AirPlay 自动同步

目标不是找一个永久 delay，而是持续判定“当前 route 状态是否与已知 baseline 一致”，只在重复证据同意时做小步修正。

验收：
- App 内需要显式 `syncContext` 状态机，而不是只依赖外部脚本：`valid / suspect / measuring / readyToDryRun / applied / locked`。
- AirPlay connection state、active set、volume/mute、device appeared/disappeared、mode enter whole-home、capture backend、sleep/wake、sidecar/OwnTone restart、local FIFO lag/drop/reconnect 任一变化都必须 bump `airplayTimingEpoch`、标记 `suspect`、取消 pending apply。
- `suspect` 后先等待 10-30s settle，再跑 no-mic/no-probe readiness；如果没有麦克风权限或 Whole-home/socket 不 ready，只显示 baseline invalid / needs passive evidence，不假装固定 delay 仍可信。
- 被动采集只在 `passive_status` preflight 通过后打开麦克风。
- 不播放 probe，不注入 pilot，不自动写 delay。
- 证据必须绑定 route context、current delay、delay lock、enabled/active AirPlay count、AirPlay timing epoch、capture backend、`syncContextState`、`syncContextRevision`。
- waveform NCC 与 envelope correlation 必须在 aggregate 和 overlapping windows 上一致。
- 漂移斜率、强多路径、上下文变化、样本尾部缺证据时 fail closed。
- 自动 apply 只能在两次独立、同 baseline、同方向、同上下文的小步 correction 后进入 dry-run；真正写入仍需单独放行。
- 多 AirPlay receiver 目前只做 group-level 诊断和 UI 提示；共享 PCM 下不能声称 per-receiver 自动修正。

### P2：DRM-safe capture 方向

Stereo 默认走 Direct Stereo；capture-dependent 路径继续推进 Process Tap，SCK 只作为 fallback。

验收：
- Netflix / Apple TV+ / Amazon / Disney+ 播放时，本地 Stereo 不被 SyncCast 触发的 SCK 阻断。
- Tap smoke 不能被 Direct Stereo 偶然绕过；必须显式 `SYNCAST_STEREO_PATH=capture`。
- Tap 在 AirPlay / passive capture 路径上有非零 callback、frames、peak 证据。

## 3. 本轮实现增量

- `passive_delay_estimator.py` 的 CLI 默认改为 `dual`。
- dual estimator 从“最终结果接近”升级为“waveform/envelope aggregate delay 接近，并且同一批 accepted windows 也要接近”。后续迭代还把 default dual mode 的 `path_candidates` 改成 waveform/envelope 都支持的候选；waveform-only 反射仍会出现在 aggregate peak diagnostics 中，但不能直接进入 Local/AirPlay path-pair correction evidence。
- feature disagreement 现在受 `max_mad_ms` 约束；默认 capture 的 25ms feature delta 不再能绕过 10ms MAD 风险。
- drift slope 不再用中位数抵消相反方向的 feature slope；下游 gate 会看到 worst-absolute slope。
- `passive_capture_estimate.py` / `passive_drift_monitor.py` 对新参数做了兼容默认值，避免手工 Namespace 或旧调用方崩溃。
- `passive_capture_estimate.py` 增加 `--report-path`；`passive_drift_session.sh` 现在会在 session 目录里同时写 `capture_preflight.json` 和 drift `preflight.json`。
- `passive_session_audit.py` 现在把 `capture_preflight.json` 当成独立安全门；如果 capture preflight 或 drift preflight 已经 `capture_failed`，不会再把缺少 monitor/samples/summary/decision 记成额外问题，避免把“现场未就绪”误读成“后续 artifact 丢失”。
- `passive_control_report.py` 现在向最终报告透出 `capturePreflightJson` / `capturePreflightVerdict`，方便区分“采集链路没 ready”和“后续 drift/correction 链路损坏”。
- auto-start 路径现在也有独立的 `auto_start_capture_preflight.json` 和 `auto_start_preflight.json`。如果 `SYNCAST_PASSIVE_AUTO_START_TARGETS` 启动后 Whole-home/socket 仍未 ready，audit/control report 会停在 `auto_start_capture_preflight` 或 `auto_start_preflight`，不会误报正常 session 的 `capture_preflight.json` 缺失。
- `control_report.json` 增加 `blockingStage` 和 `nextAction`。自动化可以直接读出当前卡在 auto-start、capture preflight、baseline、repeat confirmation、apply dry-run 等哪个阶段，以及下一步应做什么。
- auto-start manifest 现在显式记录诊断副作用：`launchesApp`、`changesRoutes`、`changesLaunchEnvironment`、`mayChangeDefaultOutput`、`changesDefaultOutput`、`defaultOutputReport`。`passive_session_audit.py` 会拒绝缺少这些声明的 auto-start corpus；`passive_control_report.py` 会把这些字段透出。这样以后自主跑 Logitech corpus 时，不会把一次改了默认输出/路由的 session 误读成完全无副作用的证据。
- auto-start setup 现在是独立 artifact：`auto_start_setup.json`。CoreAudio default-output probe 失败、LaunchServices 无法打开 app、direct executable 启动失败等早期问题会停在 `blockingStage=auto_start_setup`，不会误报成 manifest 不可信或后续 capture artifact 缺失。2026-05-16 在当前 Codex shell 里尝试 Tap/no-probe live run 时，CoreAudio helper 返回 `default output read failed outputOSStatus=0 systemOSStatus=0`；允许 unverified default 继续后，`open /Applications/SyncCast.app` 返回 LaunchServices `kLSNoExecutableErr`。随后新增 `SYNCAST_PASSIVE_AUTO_LAUNCH_MODE=exec` fallback，直接执行 `/Applications/SyncCast.app/Contents/MacOS/SyncCastMenuBar`，但 AppKit 在 `HIServices/_RegisterApplication` 阶段 `Abort trap: 6`，因此仍未采到 live Logitech corpus。最新 artifact `/private/tmp/syncast-passive-live-tap-exec-tail-20260516-codex/control_report.json` 正确记录 `launchMethodRequested=exec`、`launchMethodUsed=exec`、`appLaunchAttempted=true`、`appLaunched=false`、`defaultOutputReadFailed=true`、`defaultOutputVerified=false`、`directLaunchStderrTail`，且没有打开麦克风/发声/写 delay。
- 新增 `SyncCastPassiveHeadless` SwiftPM executable（`core/router/Sources/SyncCastPassiveHeadless/main.swift`）。它不进入 SwiftUI/AppKit，直接启动 sidecar、发现 CoreAudio/AirPlay 设备、按 `SYNCAST_HEADLESS_TARGETS=display,xiaomi` 选择本地+AirPlay 输出、进入 Whole-home，并绑定 no-probe passive diagnostic socket；active probe RPC 保持关闭，默认不允许真实 delay apply。构建通过：`swift build --package-path core/router --product SyncCastPassiveHeadless`。但当前 Codex shell smoke 仍被系统环境挡住：`SyncCastPassiveHeadless --list-devices` 返回 AirPlay browser `ServiceNotRunning` 且没有 CoreAudio device，`--targets display,xiaomi --settle-sec 3` 失败为 `missing=display,xiaomi; seen=`。这说明 headless 已绕开 AppKit 启动崩溃，但当前 shell/coalition 仍不能作为真实 CoreAudio/AirPlay 现场 runner。
- `passive_drift_session.sh` 现在支持 `SYNCAST_PASSIVE_AUTO_LAUNCH_MODE=headless`，会自动构建/启动 `SyncCastPassiveHeadless`、设置 `SYNCAST_HEADLESS_TARGETS`、`SYNCAST_HEADLESS_SOCKET`、`SYNCAST_CAPTURE_BACKEND`，并把 `launchesHeadlessRuntime`、`headlessRuntimeLaunched`、`headlessLaunchStderrTail` 写入 manifest/setup/control report。safe-fail 证据：`/private/tmp/syncast-passive-headless-autostart-20260516-codex3/control_report.json`，无麦克风/无发声/无 delay 写入，`blockingStage=auto_start_capture_preflight`，stderr 清楚记录 AirPlay browser `ServiceNotRunning` 与 `missing=display,xiaomi; seen=`；`readiness.json` 的 next action 现在正确指向 `SyncCastPassiveHeadless` 而不是菜单栏 app。
- `SyncCastPassiveHeadless --list-devices` 现在尊重 `--settle-sec`，并在 discovery 前记录直接 `CoreAudioDiscovery().enumerate()` 结果。2026-05-16 复测：普通 Codex shell 与 `launchctl asuser 501` 下均为 `direct CoreAudio enumerate returned 0 output devices`，AirPlay browse 仍为 `ServiceNotRunning`，5s discovery list 仍为 0；AppleScript `do shell script` / `display dialog` 也因当前会话 Standard Additions 不可用而无法作为替代 launcher。结论：当前阻塞是 Codex 子进程/coalition 的 CoreAudio/Bonjour 可见性边界，不是 headless list 等待太短，也不是主动 probe 或同步算法本身。
- AppModel 现在有最小运行时 `syncContext` 状态：`valid / suspect / measuring / readyToDryRun / dryRunReady / applied / locked`。进入 Whole-home、设备 toggle/reconnect/disappear、AirPlay endpoint/connection state 变化、音量/mute 变化、手动 delay 变化、delay lock/unlock、wake 事件、local FIFO broadcaster stop/restart、bridge client 数变化、overflow/per-client drop 增加、reported FIFO delay 异常变化都会让 Local+AirPlay baseline 进入 suspect/locked；active/diagnostic apply 成功会标记 applied；app-owned passive autosync 的 `dry_run_ready` 结果只有在 no-audio/no-delay-write 且 runtime echo 仍匹配时才会提升为 `dryRunReady`，表示候选已被 app-side dry-run 接受但仍未写入 delay。App 现在还保留结构化 dry-run candidate runtime record（target/current delay、context signature、capture backend、enabled+active AirPlay count/epoch、accepted-from state/revision、accepted revision、session root/control report、accepted timestamp），并通过 `passive_status` / readiness 透出，避免后续控制策略只能解析 `syncContextReason`。这不是自动同步完成，而是把“当前 delay 是否可信/是否已有安全候选”从外部脚本概念推进到 app 内状态机。
- Whole-home wake handler 不再只记录“AirPlay 自愈”。Stereo wake recovery 保持原有 `forceLocalDriverRebuild` 路径；Whole-home wake 现在会标记 syncContext suspect、调用 `router.noteWholeHomeTimingInstability` bump `airplayTimingEpoch` 并清掉 stale tau cache，然后在 running + 有启用路由时触发一次 `reconcileEngine()` / `reconcileBackgroundCalibration()`，重推 AirPlay active set、重建/更新本地 bridges、重装 passive diagnostic socket。这个路径不打开麦克风、不发 probe、不写 delay，但能避免 wake 后继续复用旧 baseline。
- 2026-05-16 追加 local bridge resync invalidation：Router 现在把 `LocalAirPlayBridge` 的 `driftResyncCount/reason/frameDelta` 暴露成结构化 diagnostics；AppModel 在 Whole-home 轮询时如果看到已有 bridge 发生新的 render resync，会把 `syncContext` 标记为 suspect，并调用 `noteWholeHomeTimingInstability` bump `airplayTimingEpoch`、清 AirPlay tau cache。这样本地 bridge 因 underrun/overrun/drift 发生相位跳变时，不会继续复用旧的 Local+AirPlay baseline。
- 2026-05-16 追加 active calibration internal volume suppression：外部/用户 AirPlay 音量变化仍会 bump `routeMutationRevision` / `airplayTimingEpoch` 并清 AirPlay tau cache；但 active calibration 自己为了本地阶段和 AirPlay 阶段而 mute/restore AirPlay 的内部音量操作不再污染自己的 route snapshot。这避免 full diagnostic calibration 在没有外部路由变化时被自己的 mute/restore 误判成 route mutation。
- 2026-05-16 追加 active/continuous delay freshness hardening：Continuous active loop 如果真的通过 Router 写入新的 delay，AppModel 不再只同步 slider/UserDefaults，而是同步把 `syncContext` 标记为 `applied` 并推进 revision。`calibrate_apply` diagnostic RPC 在写入前也会重新检查 current delay、route context、enabled/active AirPlay count、AirPlay timing epoch、syncContext state/revision；任何变化都会 fail closed，不再只靠 context signature/current delay 两个字段判断。
- 2026-05-16 追加 AirPlay repeated-connected relock invalidation：Router 现在把 active AirPlay receiver 的 repeated `connected -> connected` 事件视为可能的 AirPlay 会话重锁/输出重选，仍会 bump `airplayTimingEpoch` 并清 AirPlay tau cache；普通状态变化也统一清 cache。这样 sidecar/OwnTone 重新确认同一个 receiver 已选中时，不会因为 UI 状态没变就继续复用旧 timing baseline。
- 新增 `passive_readiness_report.py`，并由 `passive_drift_session.sh` 写出 `readiness.json`。它只读检查 app 是否安装、`SyncCastMenuBar` 是否运行、diagnostic socket 是否存在、ping/passive_status 是否 ready；不打开麦克风、不发声、不改路由、不应用 delay。它支持 `--wait-sec` / `--interval-sec`，会记录 `attemptCount` / `attempts` / `waitedSec`，供 auto-start 或外部自动化等待 Whole-home/socket readiness。
- `passive_readiness_report.py` 现在把 `syncContextState` 翻译成机器可读下一步：`passiveEvidenceIntent=baseline_required / drift_monitor / diagnostic_locked / post_apply_validation / dry_run_candidate / manual_validation_required`，并输出 `passiveEvidenceIntentSource=app_status|derived`、`baselineRequired`、`passiveCanApply`、`recommendedWorkflow`、`recommendedSessionMode`、`requiresBaselineStore`、`allowsPassiveApply`。`manual_validation_required` 会复制 dry-run candidate 字段到 readiness 顶层，包括 active AirPlay count；app runtime 返回未知 `passiveEvidenceIntent` 时现在 fail closed，不再猜成 `monitor_drift`。`passive_workflow_guard.py` 现在会在这个 workflow 下阻止新麦克风 corpus，`passive_autosync_controller.py` 则可规划一次 no-mic/no-audio/no-delay 的 accepted-candidate guard dry-run。新增 `scripts/passive_apply_accepted_candidate.py` 调用新的 `passive_apply_accepted_candidate` RPC；该 RPC 默认 dry-run，非 dry-run 必须匹配 app-stored accepted candidate、live `dryRunReady` revision、current delay、route context、capture backend、AirPlay enabled+active count/epoch 和 max-step guard；helper 也要求 readiness/result 回声绑定 `activeAirplayCount`，避免 AirPlay 临时断连/重连后应用旧 dry-run 候选。2026-05-16 最新 hardening 还把 controller 刚接受的 session/control-report/context identity 作为 `--expected-*` 传给 helper，防止下一步 apply 意外写入另一个后来的 accepted candidate。`passive_control_report.py` 透出这些字段为 `readinessPassiveEvidenceIntent`、`readinessPassiveEvidenceIntentSource`、`readinessBaselineRequired`、`readinessPassiveCanApply`、`readinessRecommendedWorkflow`、`readinessRecommendedSessionMode`、`readinessRequiresBaselineStore`、`readinessAllowsPassiveApply`。这让下一次 live Logitech corpus 可以直接知道是录当前 route baseline、做 locked diagnostic、验证刚 applied 的 delay、运行 app-side dry-run，还是停在已 dry-run 接受/需要显式人工或产品策略确认的状态，并区分“桌面 app 权威状态”和“Python fallback 推理”，而不是人工解释 readiness JSON。
- Swift diagnostic socket 的 `passive_status` 现在也直接输出同一组 runtime intent 字段：`passiveEvidenceIntent`、`baselineRequired`、`passiveCanApply`、`passiveNextAction`、`passiveEvidenceReason`。Python readiness 会优先采用 app 运行时提供的字段，只在旧 runtime 缺少字段时 fallback 推理。这样权威状态来自桌面会话内的 SyncCast，而不是外部 shell 重新猜测。
- `passive_status` / `readiness.json` / `control_report.json` 现在透出 `syncContextState`、`syncContextReason`、`syncContextRevision`。2026-05-16 追加的实现把它从“只显示”升级为安全门：Swift `passive_capture` 在打开麦克风前要求完整 route/sync context；capture start/end 会检查 `syncContextState` 和 `syncContextRevision` 没有变化；`passive_capture_estimate.py`、`passive_drift_monitor.py`、`passive_delay_decision.py`、`passive_baseline_store.py`、`passive_correction_gate.py`、`passive_apply_candidate.py`、`passive_control_report.py` 都把 sync context 作为 first-class 运行时证据/apply 绑定；baseline store 会记录 baseline 采集时的 sync context，但 baseline 身份只由稳定 route/backend/delay-lock/AirPlay group/epoch 决定。App 侧 `PassiveApplyGuard` 在 dry-run/apply 前拒绝 missing/changed/locked/measuring/dryRunReady sync context，避免已完成 dry-run 的旧候选被再次拿来写 delay。这仍不是自动同步完成，但已经关闭 stale passive candidate 覆盖新 route 状态的主要漏洞。
- 2026-05-16 追加 no-probe baseline 回填路径：新增 diagnostic RPC `passive_mark_baseline_valid`、Swift `PassiveBaselineMarkGuard`、`scripts/passive_mark_baseline_valid.py` 和 session artifact `passive_baseline_mark.json`。当 audited passive session 记录了第一条可信 baseline 后，`passive_drift_session.sh` 默认用 `SYNCAST_PASSIVE_BASELINE_MARK_MODE=mark` 调用 app-side guard，把当前 Local+AirPlay `syncContext` 从 suspect/applied 标成 valid；该 RPC 不发声、不打开麦克风、不写 delay，并在 route signature、current delay、delay lock、enabled/active AirPlay count、AirPlay timing epoch、capture backend、sync context state/revision 任一变化时拒绝。baseline mark 会把 runtime revision 从 suspect/7 推到 valid/8 这类新状态，因此 `passive_baseline_store.py` 不能把 sync context 放进 baseline key；最新实现只用稳定 route identity 找 baseline，同时让 correction gate / apply request 使用当前 decision features 里的最新 sync context。`dry-run` / `off` 模式可用于验证或禁用。`passive_control_report.py` 同时绑定 `baseline_recorded` finalize artifact 到当前 monitor features，并校验任何 `passive_baseline_mark.json` 的 session root、request fields、dry-run state、no-audio/no-mic/no-delay flags 与 runtime result，防止旧 baseline/mark artifact 误触发或误报 runtime valid。`marked_valid` artifact 现在满足默认 `baselineMarkMode=mark` 要求并归类为 OK 的 `baseline_recorded`；`baseline_mark_dry_run_ready` 有明确 `nextAction`，并作为显式 dry-run 成功状态返回 OK exit code，便于外层自动化区分“真实 mark 已完成 / dry-run 接受 / 缺 artifact 或失败”。
- 2026-05-16 追加 baseline mark artifact hardening：`passive_mark_baseline_valid.py` 在写 artifact 前会拒绝 accepted/applied 但 runtime echo 与 request 不一致的 RPC 回复，包括 route/current delay/backend、enabled/active AirPlay count、AirPlay timing epoch、no-audio/no-mic/no-delay safety flags、dry-run state、dry-run syncContext echo，以及 mark 模式的 `previousSyncContextState/Revision` 和新的 valid revision。`passive_control_report.py` 也要求 `marked_valid` artifact 的 result 显式回显 `previousSyncContextState/Revision` 且与 request 一致；当 request 从非 `valid` 状态标到 `valid` 时，result 的 `syncContextRevision` 必须前进。这样旧 artifact 或 stale RPC reply 不能只伪造 `syncContextState=valid` 就被当成 app runtime 已完成 baseline 回填。
- 2026-05-16 review 后追加 apply/gate artifact result 校验：`passive_apply_candidate.py` 在写 artifact 前会拒绝 `wouldApply/applied=true` 但 result runtime echo 与 request 不一致、dry-run 却 applied、apply 模式只 wouldApply 未写、或 `appliedDelayMs` 不等于 target 的 RPC 回复；`passive_control_report.py` 也会校验 `correction_gate.json` 属于当前 session root，并校验 `passive_apply.json` 的 request 与 app RPC result 是否都等于 gate，包括 target/current delay、route context、enabled/active AirPlay count、AirPlay timing epoch、capture backend、`syncContextState` 和 `syncContextRevision`。这防止“request 是新的，但 gate/result 来自旧上下文”的 false-ready 报告。
- 2026-05-16 追加 app-side passive apply backend hardening：Swift `PassiveApplyGuard` 现在要求 apply candidate 显式绑定 supported capture backend（`tap` 或 `sck`），runtime backend 也必须是 supported backend；缺失、空值或 `unknown` backend 会 fail closed。这样 app-side dry-run/apply 不会在 passive capture backend 不明确时返回 `wouldApply=true`。

### Passive correlation 默认算法边界

- 一次 cycle 抓 4-8s 正常节目 reference PCM + Logitech mic PCM；不发 probe，不静音/压低节目，不写 delay。
- 双特征估计：waveform NCC + envelope correlation；aggregate delay 与 overlapping windows 都必须一致，差值上限约 20-25ms。
- 每 2s window / 1s hop；reference RMS、NCC score、peak z-score、prominence、dominant cluster fraction 都要过门。
- 单 cycle dominant cluster 半径约 10ms、MAD <= 10ms；跨 cycle 3 次取 2 次同 cluster，range <= 20ms、MAD <= 15ms。
- 第一次稳定 report 只建立 `observed_offset = measured_delay - currentDelayMs` baseline；后续同 route context、capture backend、AirPlay timing epoch、enabled/active AirPlay count、`delayLocked=false` 才能找到同一 baseline。baseline 记录时的 `syncContextState/Revision` 是证据，不是 baseline 身份；当前 session 的 `syncContextState/Revision` 必须作为 correction gate / dry-run / apply 的运行时 freshness guard。

## 7. 23:43 现场复测后的需求修订

本轮在用户设备保持在线时从 Codex 重新尝试 Local + AirPlay 自动同步。系统默认输出先从 macOS `多输出设备` 切到普通 `PG27UCDM`，避免测试同时叠加一个不可控的系统 Multi-Output timeline。Tap auto-start 成功打开 SyncCast、进入 Whole-home，并选中 `PG27UCDM`、`MacBook Pro扬声器`、`Xiaomi Sound-6853`；Xiaomi AirPlay receiver 成功 connected。但 180 秒 readiness window 内 Tap capture 一直是 `backend=tap seen=0 written=0 ticks=0 peak=0/0 last=started`，AirPlay writer 和本地 bridges 都在 underrun/silence 路径上推进。因此本轮没有打开麦克风、没有采集 corpus、没有发 probe、没有写 delay。

SCK 对照复测也失败在权限/DRM 风险链路：日志显示 `screen-recording preflight: false`、`screen-recording status: notDetermined`、`router.start FAILED: screen recording permission denied`，随后 diagnostic socket 不出现。SCK 仍只能作为手动授权后的 fallback 验证路径，不能作为默认产品方向；Local Stereo / DRM-safe 路线仍应优先 Direct Stereo 和 Process Tap。

由此新增 P0/P1 实施要求：

- `passive_status` 不能只看 backend 名称与 route ready；如果 capture reference 没有收到系统音频帧，`passiveCaptureAvailable` 必须 fail closed。Swift diagnostic server 与 Python readiness/capture preflight 都要拒绝 `captureTickCount` 缺失、非整数或 `<=0`，并且在打开麦克风前失败。
- app-owned passive autosync 不能只依赖 `pgrep -x SyncCastMenuBar`；菜单栏 app 启动 bundled controller 时必须传入自己的 PID，readiness 脚本要用这个 PID 作为辅助 liveness 证据，避免“app 明明在运行但 controller 规划成 process missing / no executable command”。
- Whole-home 启动和运行中 reconcile 后必须把当前 `airplayDelayMs` 重新推给 sidecar `local_fifo.set_delay_ms`。用户手动或持久化 delay 的意义是“本地 FIFO delay line 当前实际应用了这个值”；如果 sidecar broadcaster 使用默认值或重启后没重新应用，Local + AirPlay 听感会随机漂移。
- Active coded probes 继续视为 audible-risk lab diagnostic，不得作为默认自动同步路线。当前 `comfort-21k` 的 24ms symbol grid 和多载波差频在真实音箱/DSP 上仍可能产生可闻低频重复感；默认产品路径仍是 passive/no-probe。
