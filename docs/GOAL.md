# SyncCast Goal Loop

> Last updated: 2026-05-16
> Purpose: keep future Codex/agent sessions aligned on the real product goal, not only the latest implementation thread.

## Product Goal

SyncCast should become a reliable macOS menubar app for routing system audio to more than one output without breaking normal media playback.

Reliability means the user can leave it running during normal Mac use, sleep/wake, device changes, DRM video playback, and long listening sessions without babysitting routing, reselecting devices, or quitting SyncCast.

## Current Truth

## Product Requirements

### P0: Protect the working Stereo core

Stereo local-output mode is the currently useful product. It is the baseline every future capture, routing, or AirPlay change must preserve.

Acceptance:
- Two or more local CoreAudio outputs can play in sync.
- Sleep/wake recovery works after natural display sleep, not only `pmset displaysleepnow`.
- Device identity is persisted and reconciled by stable CoreAudio UID, never transient `AudioDeviceID`.
- Stereo remains complete and usable even if AirPlay is disabled, hidden, or broken.

### P1: Remove SCK from the default Stereo path

Replace ScreenCaptureKit for always-on media playback with two non-SCK paths:

- **Direct Stereo Output:** create/manage a public CoreAudio aggregate or multi-output device and make it the system default output for local Stereo. This avoids screen capture and system-audio capture entirely for the DRM-critical local mode.
- **Core Audio Process Tap:** continue the `SystemAudioCapture -> RingBuffer` architecture for capture-dependent paths such as AirPlay, calibration, and future routing.

Acceptance:
- Direct Stereo Output can drive two local outputs without requesting Screen Recording or System Audio Recording.
- DRM playback continues while SyncCast is running in the default Stereo path.
- Process Tap mode does not request Screen Recording and keeps SCK as fallback only.
- Tap still feeds the existing `RingBuffer` contract: 48 kHz, stereo, Float32 planar.

### P2: Rebuild AirPlay as a measured experimental system

The immediate Whole-home fix is Local + AirPlay alignment control. The slider must actually push the local bridge delay (`local_fifo.set_delay_ms`) so the user can tune local playback against AirPlay playback.

After that, Whole-home / AirPlay must still not be described as solved until it passes long-session sync and recovery gates.

Strategy:
- Split AirPlay into **AirPlay-only remote group** and **Local + AirPlay mixed mode**.
- Treat AirPlay receivers as the high-latency clock domain. Local playback in mixed mode must be delayed to the AirPlay group floor; it cannot stay at Stereo's low-latency target.
- Use per-device offsets only for stable, measured bias. Do not expect a manual slider to fix drift, jitter, receiver restart, or group re-lock.
- Use passive no-probe sound-field evidence first to estimate **Local vs AirPlay group** offset. Active probes are explicit lab diagnostics only. Per-AirPlay-device acoustic identity requires TDMA/muting or a future per-receiver stream architecture.
- Do not promote passive evidence to automatic apply until the new reference/microphone timing metadata has been verified in a real Logitech capture; a stable biased passive baseline would be worse than no baseline.

Acceptance before calling it reliable:
- Local + AirPlay delay changes are audible, persisted, and logged as `airplayDelay applied: ...ms`.
- At least two AirPlay receivers play the same stream for 2+ hours without manual re-sync.
- Measured skew, drift, receiver state, buffer health, and resync events are logged over time.
- Sleep/wake, network drop, receiver restart, sidecar restart, and OwnTone restart all have explicit recovery behavior.
- The UI surfaces AirPlay as experimental until the above passes repeatedly.

### P3: Build a measurement loop

Manual delay helps first-run perception, but it does not solve long-session reliability.

Acceptance:
- Add a repeatable AirPlay sync protocol and log format.
- Record per-device latency, drift, packet/buffer health, OwnTone output state, and local bridge lag.
- Decide based on evidence whether to keep OwnTone, replace the AirPlay sender, or defer AirPlay behind the stable Stereo product.
