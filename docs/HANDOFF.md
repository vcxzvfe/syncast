## 1. What SyncCast is

macOS open-source menubar app for routing system audio to multiple outputs. Default local Stereo no longer captures system audio; capture-dependent fallback and AirPlay paths may still use ScreenCaptureKit unless Process Tap is explicitly selected.

- **Stereo (local) mode** — multiple local CoreAudio outputs (built-in speaker, USB DAC, HDMI monitor speakers). The default path is Direct Stereo Output, which creates/manages a public CoreAudio multi-output style aggregate and makes it the system default while active; the old private AggregateDevice + AUHAL render path remains available only with `SYNCAST_STEREO_PATH=capture` / `sck`.
- **Whole-home (AirPlay) mode** — experimental local CoreAudio + AirPlay 2 receivers via OwnTone broadcaster + sidecar/local bridge path. The near-term sync target is local CoreAudio speakers vs. the AirPlay group. AirPlay receiver-to-receiver timing should be treated as AirPlay's own buffered clock domain unless future evidence says otherwise.
- **Process Tap capture** — experimental `SYNCAST_CAPTURE_BACKEND=tap` backend under the shared `SystemAudioCapture` interface; live non-DRM smoke passed on this machine, but DRM and sleep/wake validation remain pending.

Modes are mutually exclusive. Switching tears down + rebuilds the local driver stack.

Read `docs/GOAL.md`, `docs/requirements_2026-05-16.md`, and `docs/goal_completion_audit_2026-05-12.md` before planning new work. They capture the current product truth: protect Stereo local mode, replace SCK for DRM compatibility via Direct Stereo Output + Process Tap, and treat AirPlay as experimental until measured over long sessions.

## 2. Architecture (file:line map)

### Core router (`core/router/Sources/SyncCastRouter/`)
- `Router.swift` — top-level actor. `start(devices:)` / `stop()` / `setRouting(...)` / `forceLocalDriverRebuild(devices:)` / `runCalibration(...)`. Owns the selected capture backend, local aggregate/direct output, audio socket writer, and OwnTone broadcaster.
- `SystemAudioCapture.swift` — shared capture backend interface.
- `SCKCapture.swift` — SCStream system-audio capture fallback/default for capture-dependent paths when Process Tap is not selected. Writes into `RingBuffer`. Has `onUnexpectedStop` callback the Router wires up.
- `TapCapture.swift` — Core Audio Process Tap prototype behind `SYNCAST_CAPTURE_BACKEND=tap`; installed live non-DRM smoke passed locally with non-SyncCast program audio, while DRM playback, sleep/wake, and AirPlay/calibration replacement validation remain pending. A 2026-05-12 Tap calibration attempt produced zero callbacks without external audio, so do not treat Tap as an SCK drop-in for AirPlay yet.
- `DirectStereoOutput.swift` — default non-capture local Stereo path; `SYNCAST_STEREO_PATH=capture` / `sck` select the old capture-backed fallback.
- `AggregateDevice.swift` — private CoreAudio aggregate device (UID `io.syncast.aggregate.v1.<pid>.<uuid>`). `pickMaster()` always picks built-in (transportType=BuiltIn=score 0) over HDMI (=50).
- `LocalOutput.swift` / individual AUHAL render path.
- `LocalAirPlayBridge.swift` — local CoreAudio bridges that consume from sidecar's audio socket (whole-home mode).
- `CalibrationTransportHealth.swift` — pure transport-counter health gate used to fail acoustic calibration closed when writer/bridge state changes during measurement.
- `Capture.swift:137-159` — `deviceID(forUID:)` resolves UID → AudioDeviceID at runtime via `kAudioHardwarePropertyTranslateUIDToDevice`.
- `ActiveCalibrator.swift` — active acoustic calibration: continuous-phase high-band coded FSK probes, repeated local/AirPlay group cycles, dominant tau cluster selection, confidence/MAD/range/slope gates.

### Discovery (`core/discovery/Sources/SyncCastDiscovery/`)
- `CoreAudioDiscovery.swift:37-90` — enumerates CoreAudio outputs, listens to `kAudioHardwarePropertyDevices`, emits `.appeared / .updated / .disappeared`.
- `Device.swift:20` — `coreAudioUID: String?` is the stable identifier across sleep/replug. NEVER use `AudioDeviceID` (UInt32) for persistence.
- `StableIDMap` — process-internal `"ca:<UID>"` → SyncCast UUID hash.

### Sidecar (`sidecar/`)
- Python (PyInstaller-bundled). RTSP/AirPlay glue around `pyatv`. JSON-RPC over Unix socket.
- `audio_socket.py` — `LocalFifoBroadcaster` with 30 ms ramped delay (Round 11).

### Menubar app (`apps/menubar/Sources/SyncCastMenuBar/`)
- `SyncCastApp.swift` — `MenuBarExtra`. `statusIcon(name:)` helper unifies SF Symbol vs Bundle.module loose-PNG (Round 11).
- `AppModel.swift` — `@Observable @MainActor`. Owns `routing`, `mode`, `streamingState`, sleep/wake observer (Round 12), Local+AirPlay `syncContext`, and the app-owned no-probe passive autosync runner. The runner has both manual `Passive Check` and automatic scheduling after suspect sync-context events; scheduled runs and passive mark/apply RPC callbacks bind to the current `syncContextState/Revision` before mutating app state. Manual `Passive Check` can request first-time microphone permission and continue into the same run after grant; automatic background runs never prompt and require already-authorized mic permission. Each passive run has a UUID runID and unique artifacts so canceled/stale tasks cannot clobber a newer run. Cancel keeps the UI busy in `Stopping` until the child controller exits, with a same-run force-kill fallback after 5s. Critical methods: `bootstrap()`, `reconcileEngine()`, `handleWake(notification:)`, `startPowerEventWatch()`, `applyEvent(_:)`, `runPassiveAutosyncOnce()`.
- `MainPopover.swift` — SwiftUI popover. AirPlay manual delay slider (Round 11) plus the real `Passive Check` / `Grant Mic` entry for app-owned no-probe passive measurement. There is still no product button literally named `Auto Calibrate`; active tone tools are lab-gated under Diagnostics.
- `Resources/MenubarIcon.png` (+@2x +@3x) — flat PNGs (no xcassets, codex caught the build problem in Round 11).
- `Resources/AppIcon.icns` — 612 KB built from `assets/branding/app-icon-1024.png`.

- `scripts/passive_capture_estimate.py` — multi-cycle no-probe passive evidence harness. It pings the diagnostic socket before capture, then requires a hard `passive_status` readiness gate before every capture cycle: `ok`, `passiveCaptureAvailable`, `inProgress == false`, supported backend (`sck`/`tap`), enabled and active AirPlay counts, current FIFO delay metadata, delay-lock metadata, AirPlay timing epoch, and route context. Accepted cycles must also contain valid passive mic timing metadata (`sampleRate`, `microphoneArmedAtNs`, `microphoneFirstSampleAtNs`, `microphoneStartPaddingFrames`, `microphoneWarmupFramesDropped`) and unchanged AirPlay timing epoch with padding consistent with arm-to-first-sample timing. Single-cycle runs are diagnostic-only unless explicitly overridden; ambiguous strong multi-peak evidence fails closed. It does not apply `airplayDelayMs`.
- `scripts/passive_capture_estimate_tests.py` — offline tests for the passive readiness gate, inactive AirPlay rejection, passive consensus gate, cycle-level context-change rejection, mid-capture context-change rejection, preflight/capture context mismatch rejection, missing/inconsistent mic timing metadata rejection, incomplete-reference rejection, bimodal cycle rejection, and strong-peak summary.
- `scripts/passive_drift_monitor.py` — no-probe long-session passive monitor. It repeatedly runs the passive consensus harness over real program audio, summarizes accepted delay drift, and exits `0` stable / `3` inconclusive / `4` unstable / `2` capture failed. It fails closed on trailing inconclusive samples by default, so a front-loaded stable section cannot hide missing evidence near the end. It also refuses to report stable if accepted samples change route context, applied delay, enabled or active AirPlay count, AirPlay timing epoch, capture backend, or delay-lock state. Default capture estimation is dual-mode and can reject excessive within-capture drift slope when `--max-estimate-slope-ms-per-min` is set. Use `--preflight-only` first to validate args, diagnostic-socket reachability, and the same no-mic `passive_status` readiness gate without opening the mic, emitting audio, or applying delay; normal monitor runs re-check that gate before every passive capture cycle. Use `--report-path` for the final JSON report and `--jsonl-path` for per-sample append-only evidence during long runs; if capture later fails, the report path still records `capture_failed` plus rows collected before the failure. It still does not launch SyncCast, emit audio, change routes, or apply `airplayDelayMs`.
- `scripts/passive_drift_monitor_tests.py` — offline tests for the passive drift summary gates, active AirPlay count context gate, per-capture readiness gate, sample-level inconclusive cycle propagation, exit-code mapping, and report/JSONL writing.

- `scripts/passive_delay_decision.py` — conservative passive control-policy prototype. It treats passive delay as relative evidence, not an absolute `airplayDelayMs` target: a stable known-good report initializes a baseline, later stable single-path reports can recommend bounded small corrections from that baseline, and unsafe/unstable/locked-delay evidence fails closed. Coherent multi-path evidence can be used only when a Local/AirPlay two-path pair appears inside the same correlation cycle, one path maps near the current local FIFO delay, the path-pair delta is stable across accepted rows, and every strong-peak/multi-path cycle is explained by such a pair. The pair selector now enumerates all same-cycle candidate pairs and picks a coherent cross-row set, so a strong reflection in the top two candidates cannot hide a lower-ranked direct Local/AirPlay pair. The passive path allows 1...8 fully connected AirPlay receivers as one AirPlay group; this calibrates local delay relative to the group and does not prove per-receiver skew. Baseline reports must match the current route context, capture backend, AirPlay count, AirPlay timing epoch, and stable `delay_locked=false` state; stored `baselinePathPairDeltaMs` is reused for later coherent relative decisions. It exits `0` for usable decisions, `3` for valid-but-not-applicable evidence, and `2` for bad input. It still does not write defaults or call Router APIs.

- `scripts/passive_correction_gate.py` — repeated-recommendation gate for future automatic apply. It consumes `finalize.json`; one eligible recommendation only records `pending_confirmation`, while a second fresh different-session same-baseline/same-current-delay/same-direction/similar-target recommendation becomes `ready_for_apply_candidate`. ### Round 12 (2026-04-28 → 04-29) — sleep/wake audio recovery
**Bug**: stereo mode after monitor DPMS sleep → audio silent, user must deselect+reselect both outputs to recover.

**Root causes** (3 layered, found across 5 codex review cycles):
1. **HDMI device identity flip** — display sleep → HDMI subdevice disappears from `kAudioHardwarePropertyDevices`; on wake reappears with same UID but new `AudioDeviceID`. Existing AggregateDevice points at dead ID → silent underrun.
2. **`reconcileLocalDriver` short-circuit** — `alreadyCorrect` returns early when enabled UID set didn't change (which it didn't on transient wake), skipping the rebuild that would fix #1.
3. **SCK stream death** — display sleep also breaks SCStream (`connectionInvalid -3805`). Even after aggregate rebuild, no source feeds the new ringBuffer → silent.

**Fix chain** (commits in order):
- `2126923` v1: NSWorkspace `didWakeNotification` + `screensDidWakeNotification` double-listen + `forceLocalDriverRebuild` bypassing short-circuit. **Insufficient — only rebuilt local driver, not SCK.**
- `fec46d1`, `a95e45b`, `ba2934c` — hardening (retry-backoff, observer dedup, single-flight task, shadow set for transiently-missing-but-user-enabled UIDs, off-by-one fix).
- `5be9206` v2: SCK `stop()` + `start()` inside `forceLocalDriverRebuild` + `onUnexpectedStop` callback hook. **Codex caught 3 races.**
- `e1175c3` v3 (post Codex Cycle 1): same-stream identity guard in `didStopWithError`, assignment-before-`startCapture()`, `forceLocalDriverRebuild` returns Bool; AppModel wake loop requires `sckOK && allResolved`. **Codex caught 2 more deeper races.**
- `cd8b8df` v4 (post Codex Cycle 2): post-`startCapture()` verify-then-act guard (throws on stream race), per-attempt device snapshot inside retry loop. This is the current Round 12 fix commit; `d955eb7` is the docs handoff commit on top.

**Status**: user reports the previous audio auto-sleep / screen-sleep recovery issue is resolved; after screen sleep, playback resumes normally in Stereo mode. Latest known Round 12 fix is `cd8b8df`; current repo HEAD is `d955eb7` because the handoff document was rewritten afterward.

### B. ScreenCaptureKit + DRM incompatibility (HIGH — user-reported)
- **User feedback (2026-04-29)**: when SyncCast is running, DRM-protected playback fails (Netflix, Apple TV+, Disney+ refuse to play because SCK = "screen recording active"). Movies don't play; some apps degrade.
- This is the SCK trade-off — system audio capture currently piggybacks on `SCStream` which requires Screen Recording permission and signals to playback DRM systems that the screen is being captured.
- **Updated status (2026-05-16)**: default local Stereo now uses Direct Stereo Output and should not start SCK. The remaining work is to validate real DRM playback and keep moving capture-dependent paths away from SCK:
  - **Direct Stereo Output** — create/manage a public CoreAudio aggregate or multi-output device, set it as the system default output while SyncCast is active, and let macOS route app audio directly. Best DRM profile because SyncCast is not recording screen or system audio. Stereo-only.
  - **Audio Process Tap API** (macOS 14.2+) — `CATapDescription` lets SyncCast tap audio without Screen Recording. Keep this for capture-dependent paths such as AirPlay/calibration. It still needs real DRM validation because it is system-audio capture, even though it is not screen capture.
  - **AudioServerPlugIn** (BlackHole/Loopback-style) — fallback only if Direct Stereo Output and Tap fail. Pros: robust routing. Cons: install/uninstall UX, signing, HAL component complexity.
  - **Loopback / Rogue Amoeba ARK** — only if licensable, not a DIY path.

- See § 5 for the recommended exploration plan.

## 5. New direction — replacing ScreenCaptureKit (top priority for Codex Review)

### 5.1 Why
SCK forces Screen Recording permission AND triggers DRM playback blocks. User reported Netflix/Apple TV+ refusing to play with SyncCast running. This kills the "always-on" use case.

### 5.2 Recommended primary path for Stereo: Direct Stereo Output

For local Stereo, the best DRM profile is not to capture audio at all.

Create a public CoreAudio aggregate / multi-output device, set it as the system default output while SyncCast is active, and restore the previous default output on stop/quit. Netflix and Apple TV+ should then see only a normal CoreAudio output device.

Pros:
- No Screen Recording permission.
- No System Audio Recording permission for the direct path.
- Lowest DRM risk.
- Better video lip-sync than an AirPlay/capture buffer path.

Caveats:
- Stereo-only; does not feed AirPlay or calibration.
- Needs careful lifecycle: stale public aggregate cleanup, default-output restore, volume behavior, and crash recovery.
- Must not regress the private AggregateDevice + AUHAL path until direct mode is proven.

Suggested prototype flag: `SYNCAST_STEREO_PATH=direct`.

### 5.3 Capture-dependent path: Audio Process Tap API (`CATapDescription` / `AudioHardwareCreateProcessTap`)

Available in macOS 14.2+. Lets a process tap system-wide audio (or per-process audio) using the system-audio capture permission instead of Screen Recording. Apple specifically documents this for audio-only capture clients.

Key APIs:
- `CATapDescription` — describes what to tap (process list, mute behavior, mono/stereo).
- `AudioHardwareCreateProcessTap(...)` — returns an `AudioObjectID` you can read from like a normal CoreAudio device.
- Composes with AggregateDevice — you can include the tap as a sub-device of an aggregate.

Caveats to research:
- Permissions model — requires `NSAudioCaptureUsageDescription` and a system-audio recording grant; verify prompt/signing behavior on Sequoia/Tahoe.
- Latency vs SCK (SCK is ~10-20 ms; tap API likely lower since it's HAL-level)
- Mute behavior — `CATapMuteBehavior` controls whether the tapped process keeps hearing its own output
- Signing entitlements — confirm what's needed in `Info.plist` / code sign

Reference / exploration order:
1. Apple sample: "Capturing system audio with Core Audio taps"
2. Local SDK headers: `CATapDescription.h`, `AudioHardwareTapping.h`, `AudioHardware.h`
3. Apple WWDC/CoreAudio references as needed
4. Test reproduction of DRM playback under the new path

### 5.4 Fallback: AudioServerPlugIn / DriverKit virtual device

If the tap API doesn't deliver (DRM bypass not perfect, latency too high, or hits a Sequoia regression), fall back to a virtual audio driver:
- BlackHole-style approach: ship a lightweight DriverKit driver (or kext on older macOS) that appears as an output device. User selects it in System Settings as their default output. SyncCast reads from it.
- Pros: zero permissions UI, DRM completely invisible (just an audio device).
- Cons: significant new component, DriverKit signing, install/uninstall UX.

### 5.5 Migration plan sketch
- Phase 1: keep SCK build as fallback. Direct Stereo Output exists and is now the default local Stereo path; Tap exists behind `SYNCAST_CAPTURE_BACKEND=tap`.
- Phase 2: dogfood Direct Stereo for local playback. Normal stop/quit restore has installed smoke coverage; still verify DRM apps, abnormal exits, failed router start, user-changed default-output cases, and volume UX.
- Phase 3: dogfood Tap for capture-dependent paths. Verify: no Screen Recording prompt, AirPlay still gets PCM, Tap wake recovery works.
- Phase 4: after DRM/abnormal-exit validation, keep Direct as default Stereo, keep Tap for capture-dependent paths if it passes DRM/sleep/wake/AirPlay checks, and keep SCK as fallback only.
