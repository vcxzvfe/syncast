# Tahoe Screen Recording TCC stuck on `false` — root cause and fix

**Subject app:** SyncCast (`io.syncast.menubar`), ad-hoc signed `.app`, run from `/Users/zifan/syncast/dist/SyncCast.app`
**Symptom:** User toggles Screen Recording ON in System Settings → next process launch, `CGPreflightScreenCaptureAccess()` returns `false`. Repeats after every `tccutil reset All io.syncast.menubar`.
**OS:** macOS 26.4.1 Tahoe.

## TL;DR

This is **not** a Tahoe-specific UI bug for SyncCast's case. It is the long-standing, documented behavior of TCC + ad-hoc signing, made slightly worse on Tahoe by a separate UI regression (plain executables/unstable bundles silently disappearing from the Privacy panel). **Ad-hoc signatures cannot anchor a TCC grant across rebuilds**, and on Tahoe the failure mode is now "switch shows ON but the grant is bound to a stale CDHash, so the new process is still denied." The only durable fix is a stable signing identity; everything else is a workaround.

---

## 1. Is there a known Tahoe bug where the toggle shows ON but TCC actually has it denied?

Yes — but the precise mechanism matters. There are **two** separate Tahoe regressions and **one** pre-existing TCC behavior intersecting here:

- **Pre-existing (since at least Catalina):** TCC keys grants by code-signing identity. With an ad-hoc signature, the identity is effectively the *cdhash*, which changes on every rebuild. Apple DTS engineer Quinn "The Eskimo!" states this directly:
  > "TCC uses an app's code signature to confirm that version N+1 of your app is the 'same code' as version N. Problems like this are usually caused by your app not having a stable signing identity — either it's not signed or its ad hoc signed." ([Apple Developer Forums #695689](https://developer.apple.com/forums/thread/695689))

- **Tahoe 26.1 regression (still present in 26.4.x):** Plain executables and certain non-stable bundles request the prompt successfully, get a TCC row written, but **do not appear (or appear inconsistently) in System Settings → Privacy & Security → Screen & System Audio Recording**. Quinn confirms: *"Indeed…IMO this is a bug and I encourage you to file it as such."* ([Apple Developer Forums #807898](https://developer.apple.com/forums/thread/807898))

- **Microsoft Teams on 26.4 loop:** documented case where the Screen Recording switch reads ON, the PPPC profile says ON, the TCC.db row says ON, and the app still sees denied — Microsoft's supported remediation is *remove the entry, re-launch, re-grant*. ([Microsoft Q&A: Teams screen sharing loop on Tahoe 26.4](https://learn.microsoft.com/en-us/answers/questions/5848423/teams-for-mac-screen-sharing-permission-loop-on-ma))

For SyncCast specifically, the dominant cause is #1 (ad-hoc instability), not #2. #2 is what makes the symptom *look* like a Tahoe-only bug.

## 2. Does ad-hoc instability cause this? Does TCC pin to CDHash?

**Yes, definitively.** With `Signature=adhoc`, the signature has no certificate chain. The "designated requirement" generated by `codesign` therefore reduces to a **cdhash equality check** (see TN2206 and Eclectic Light's signing history: [eclecticlight.co — A brief history of code signing](https://eclecticlight.co/2025/04/26/a-brief-history-of-code-signing-on-macs/)). `--identifier io.syncast.menubar` sets the bundle identifier in the signature blob, but **TCC's binding is the cdhash, not the identifier**, when no team/cert chain is present. Every rebuild produces a new Mach-O → new cdhash → TCC row no longer matches the running process → preflight returns `false`.

This explains the loop: System Settings shows the *previous build's* SyncCast as enabled. The new build has a new cdhash, so TCC's lookup misses; the prompt path then writes a *new* deny/prompt row, while the old enabled row still sits there. Toggling the switch you see in Settings is updating a stale row that no longer corresponds to the running binary.

**Minimum stable signing approach:** create a self-signed Code Signing certificate via Keychain Access → Certificate Assistant → *Create a Certificate*, type **Code Signing**, *Self Signed Root*, default 365-day validity ([Apple Support guide](https://support.apple.com/guide/keychain-access/create-self-signed-certificates-kyca8916/mac)). Then sign with `codesign --force --deep --sign "SyncCast Dev" --identifier io.syncast.menubar /path/to/SyncCast.app`. This gives you a real Subject DN; the designated requirement upgrades from "cdhash equals X" to "anchor trusted AND identifier == io.syncast.menubar AND certificate leaf SubjectCN == SyncCast Dev" — stable across rebuilds. Apple's TN2206 explicitly notes that for TCC purposes, *self-signed identities work*; only Gatekeeper/notarization care whether the anchor is Apple's. ([TN2206](https://developer.apple.com/library/archive/technotes/tn2206/_index.html))

## 3. The "disappearing grant" bug on Tahoe

Manifests as: row exists in TCC.db, switch shows ON, app sees denied. Apple's only acknowledged workaround in the Teams thread and the 807898 thread is **remove the entry, relaunch, re-grant, and (for stubborn cases) reboot so `tccd` rebuilds its in-memory cache**. There is no `tccutil reset` for "purge stale cdhash rows for this bundle id" — `tccutil reset ScreenCapture io.syncast.menubar` only clears rows whose client identifier matches; rows attributed by cdhash to deleted binaries can be left orphaned.

## 4. Multiple stale entries

When several "SyncCast" rows accumulate (one per rebuild), toggling any one of them updates only the row whose stored requirement matches that historical cdhash. The currently running process is matched by **its own** cdhash; if that one isn't in the list (or is the *unticked* row), no toggle anywhere helps. Cleanup options, in order:

1. Select each row → click `−` (the minus must be visible; if greyed out, unlock with the lock icon first).
2. If `−` is missing (a known Tahoe Settings bug — [MacRumors thread 2328791](https://forums.macrumors.com/threads/fed-up-with-privacy-settings-help-needed.2328791/)), use `tccutil reset ScreenCapture io.syncast.menubar` then **reboot** to flush `tccd` and Settings' cached list.
3. Hard reset: `tccutil reset ScreenCapture` (all apps), reboot. Heavy-handed; everything else must be re-granted. ([Eclectic Light: tccutil guide](https://eclecticlight.co/2020/01/28/a-guide-to-catalinas-privacy-protection-4-tccutil/))

## 5. The "first-launch race" in `App.init()`

Calling `CGRequestScreenCaptureAccess()` from `App.init()` (SwiftUI) or from `applicationWillFinishLaunching:` is **not** known to corrupt the TCC row, but it does fire before `responsible-pid` attribution stabilises and before LaunchServices has finished registering the bundle. On Tahoe with an ad-hoc binary this can produce a row that's attributed to the *launcher* (Terminal, Finder, or `launchd`) rather than to SyncCast. The safer pattern is to call it from `applicationDidFinishLaunching:` or after `NSApp` is fully up, and to gate on `CGPreflightScreenCaptureAccess()` first so you don't redundantly trigger the prompt. (See Chromium's reference implementation: [permissions_utils.mm](https://cocalc.com/github/chromium/chromium/blob/main/ui/base/cocoa/permissions_utils.mm).)

A separate, well-known wart from Apple Developer Forums #732726: even after the grant takes effect for actual capture APIs, `CGPreflightScreenCaptureAccess()` can keep returning `false` until the **process is restarted**. Quinn acknowledges the inconsistency. ([Apple Developer Forums #732726](https://developer.apple.com/forums/thread/732726)) That alone explains why "every subsequent process launch" still reports false even when the grant actually worked — but in SyncCast's case the rebuild-induced cdhash change is the dominant factor.

## 6. Workarounds, ranked by expected success

| Rank | Action | Expected outcome |
|------|--------|-------------------|
| **A** | **Sign with a self-signed Keychain "SyncCast Dev" code-signing cert (option c).** | **Fixes the root cause.** Cdhash drift no longer matters; TCC pins to the cert + identifier. ~95% confidence. |
| B | Reboot + `−`-remove every SyncCast row + reboot again before re-granting (b). | Clears stale rows so the *current* cdhash gets a clean grant. Holds until next rebuild. ~70%. |
| C | Move `.app` to `/Applications/` (e). | Helps only marginally; TCC does inspect path stability and `/Applications` reduces quarantine/translocation risk. Won't fix cdhash drift. ~25% on its own; useful in combination. |
| D | Probe via `SCStream.startCapture` and read `SCStreamErrorUserDeclined` (d). | Diagnostic only — does not unstick the grant. Useful to confirm whether the failure is preflight-reporting-stale or genuinely denied. |
| E | Plain reboot (a). | Sometimes flushes `tccd` cache; without removing stale rows it usually doesn't help. ~30%. |
| F | Switch to a different prompt API (f). | There is no different API. `CGRequestScreenCaptureAccess` is the only documented trigger; SCK's own first-use prompt is equivalent. Not a workaround. |

## 7. Launchd / SIP / Gatekeeper / quarantine

Gatekeeper's `com.apple.quarantine` xattr matters mainly for first-run and translocation. If the `.app` was moved through a browser/AirDrop/zip, Tahoe will *path-translocate* it on launch (run from a read-only `/private/var/folders/.../AppTranslocation/...` mount), which **changes the responsible path** and breaks any TCC grant tied to the original path. Verify with `xattr -l /Users/zifan/syncast/dist/SyncCast.app` and clear with `xattr -dr com.apple.quarantine`. Building locally with `codesign` should not set quarantine, but if SyncCast was ever zipped and unzipped during distribution testing, this is worth ruling out. SIP is not implicated for the user-domain Screen Recording grant (no SIP-protected paths involved).

---

## Highest-confidence action

**Stop ad-hoc signing. Create a self-signed "SyncCast Dev" Code Signing certificate in Keychain Access → Certificate Assistant, then re-sign with `codesign --force --deep --options runtime=no --sign "SyncCast Dev" --identifier io.syncast.menubar /Users/zifan/syncast/dist/SyncCast.app`, run `tccutil reset ScreenCapture io.syncast.menubar`, reboot, then launch and re-grant.** This is the only step that addresses the actual root cause (cdhash-pinned TCC grants invalidated on every rebuild); every other workaround unsticks the symptom for one launch and reverts on the next build.
